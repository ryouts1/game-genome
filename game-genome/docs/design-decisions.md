# design decisions

## 1. TypeScript ではなく標準 ES Modules を採用

今回は zip で渡したあとにすぐ動かしやすいことを優先し、依存なしの ES Modules に寄せました。

- ビルドなしでブラウザと Node テストの両方を回しやすい
- ポートフォリオ確認者が構成を追いやすい
- 依存解決トラブルを減らせる

## 2. ルール表現は「十分に汎用」で止める

将棋・チェス・コネクト系・突破系を全部 1 本で表現したい一方、表現力を広げすぎると DSL 自体が主役になります。

この repo では、次を優先しました。

- すぐ読めること
- JSON の見通しがよいこと
- 少ないサンプルで汎用性が伝わること

## 3. AI は 2 系統に絞る

- Minimax: 小〜中規模ゲームで読み筋を説明しやすい
- MCTS: 分岐が多い配置ゲームでも対応しやすい

この 2 つがあると、採用側に「探索手法を場面で使い分ける発想」が伝わりやすいです。

## 4. 評価関数はゲーム依存ロジックを埋め込みすぎない

評価関数をゲームごとに if 文で書き分けると、JSON 駆動の価値が薄れます。

そこで次の共通特徴量に寄せています。

- material
- mobility
- centerControl
- advancement
- linePotential

ゲーム固有の差は重みで吸収する設計です。

## 5. UI は軽量でも情報密度を落とさない

ポートフォリオでは、画面の派手さより「定義を切り替えたときに何が共通で何が変わるか」が伝わることの方が重要です。

そのため UI では次を重視しました。

- 盤面そのもの
- 手番
- ルール概要
- AI の直前判断
- 手順ログ

## 6. 派生 repo を意識した構成

この repo 自体が最終成果物というより、次の派生を増やす起点として設計しています。

例:

- `quantum-shogi`
- `grid-racing-ai`
- `mini-chess-lab`
- `gomoku-search-study`

共通部分とゲーム固有部分を分けたことで、同じような CRUD を量産するよりも設計の軸が伝わりやすくなります。
