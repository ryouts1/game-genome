# architecture

## レイヤ構成

`game-genome` は大きく 3 層に分けています。

### 1. definition layer
`games/*.json` がゲーム固有の差分です。盤面サイズ、駒種、行動、勝利条件、評価重みを宣言します。

### 2. engine layer
`src/core/` が共通エンジンです。

- `definition-loader.js`: 定義の検証と正規化
- `state.js`: 初期状態生成、盤面参照、状態シリアライズ
- `move-generator.js`: 合法手列挙
- `termination.js`: 勝敗・引き分け判定
- `evaluator.js`: AI 用スコアリング
- `game-engine.js`: 状態遷移の適用

### 3. interaction layer
`src/ui/` と `src/ai/` が上位レイヤです。

- UI は現在局面を可視化し、人間操作を action に変換する
- AI は engine layer を使って探索する

## 状態遷移の流れ

1. JSON をロードする
2. `createInitialState()` で初期局面を作る
3. `generateLegalActions()` で着手候補を列挙する
4. 人間または AI が action を 1 つ選ぶ
5. `applyAction()` で次局面へ進める
6. `getGameStatus()` で終局かどうか判定する
7. 終局でなければ次の手番へ進む

## action を共通化した理由

UI / AI / 履歴ログを共通化するには、ゲームごとの着手を同じ形で扱える必要があります。

そのため現在は、着手を次の 2 種類に絞っています。

- `place`
- `move-piece`

これだけでも、三目並べ・四目並べ・突破ゲーム・小型チェス類はかなり吸収できます。

## なぜ汎用 DSL を狙いすぎなかったか

完全なゲーム DSL を最初から狙うと、ルール表現が大きくなりすぎて「何を見せたい repo なのか」がぼやけます。

このリポジトリでは、

- グリッド盤面
- 2 人対戦
- 決定論
- ターン制

という就活ポートフォリオとして説明しやすい範囲に絞り、完成度を優先しました。

## 量子将棋への接続点

量子将棋 repo を別で作る場合、再利用しやすい部分は次の通りです。

- 盤面 UI の基本構造
- 手順ログ
- ゲーム読み込みと切り替え
- AI のインターフェース
- 状態オブジェクトの管理方針

逆に、量子将棋側で追加になる部分は次です。

- 1 マス 1 駒ではない状態表現
- 観測 / 測定のイベント
- 確率や重ね合わせを考慮した評価関数
- 通常の合法手列挙ではないルール分岐
