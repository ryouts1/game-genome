# design decisions

## 1. フレームワーク側を主役にした

この repo で見せたいのは、ゲーム 1 本の完成ではなく、**ゲームを増やすための基盤設計**です。

そのため、`games/*.json` と `src/core/` の関係が読み取れることを最優先にしています。

## 2. まずは 2 人完全情報に絞った

最初から多人数・不完全情報・確率ゲームまで広げると、DSL と AI の両方が膨らみます。

今回は次を明確に切っています。

- 2 人
- 完全情報
- 決定論
- グリッド盤面
- ターン制

この制約の中で Studio と Arena まで成立させるほうが、就活用 GitHub としては価値が高いと判断しました。

## 3. ルール DSL を `place` / `move-piece` に寄せた

ルール表現を細かく増やしすぎると、説明が難しくなります。

今の DSL なら、README でも面接でも次のように説明できます。

- 配置型は `place`
- 駒移動型は `move-piece`
- 動きは `step` と `ray`
- 終局は `line` / `capture-all` / `reach-row`

この説明のしやすさを優先しました。

## 4. Studio を後付けではなく本体機能にした

この repo の価値は、定義の読み込みだけでは弱いです。  
実際に使うときは「作る・直す・比較する」が必要になります。

そのため、次を本体に入れています。

- 定義検証
- プレビュー適用
- ローカル保存
- 共有リンク
- Arena

## 5. Arena を同梱した

ゲーム定義は、実装直後に良し悪しが見えづらいです。  
特に評価関数と先手有利は、何局か回さないと感触が掴みにくいです。

そこで、別 repo にせず同梱しています。  
これにより「設計だけでなく検証手段まで用意している」ことが伝わります。

## 6. 量子将棋は別 repo に逃がせる形にした

量子将棋まで本 repo に直接入れると、状態表現が一気に重くなります。  
今回は、量子将棋に使えそうな土台だけ残しています。

- 盤面 UI
- タイムライン
- AI の入口
- 定義ローダー
- バッチ対局

その上で、重ね合わせや測定のような特殊ルールは別 repo で拡張する前提です。
